# AegisOS Technical Decisions

## 1. Communication Architecture: Protocol-based vs. Natural Language
- **Decision**: Adopt a standardized JSON protocol (AACP) combined with the Blackboard pattern, rather than pure natural language dialogue between Agents.
- **Rationale**:
  - Reduces Token consumption: Large file paths are passed via pointers, avoiding repetitive reading.
  - Determinism: JSON structure is easy for programs to process, reducing parsing ambiguity.
  - Observability: Message logs clearly reflect the system execution flow.

## 2. Security Strategy: Zero-Trust Sandbox
- **Decision**: Completely abandon local Root privileges; all Skills must run in sandboxes such as cloud WebAssembly or Firecracker VMs.
- **Rationale**:
  - Prevents host takeover caused by prompt injection attacks.
  - Implements a dispose-after-use model, isolating side effects generated by skill execution.

## 3. Interaction Method: API/CLI vs. GUI Automation
- **Decision**: Prioritize the integration of CLI tools with action translation (CLI-Anything), abandoning fragile and high-energy vision model click simulations.
- **Rationale**:
  - Improves stability: CLI output provides deterministic results.
  - Reduces latency: API interaction is much faster than headless browser rendering.

## 4. Memory Management: Hot/Cold Separation
- **Decision**:
  - **Hot Memory (RAM)**: Keep only the most recent 5 rounds of dialogue and status outlines.
  - **Cold Memory (Disk)**: Asynchronously extract facts and preferences into a Vector DB.
- **Rationale**: Solves the token explosion problem in long dialogues and achieves infinite context support.

## 5. Development Paradigm: Async-First
- **Decision**: Build the entire system based on `asyncio`, discarding traditional multi-threading.
- **Rationale**: Efficiently handles Agent waits for LLM returns and heavy file I/O tasks, reducing system resource overhead.

## 6. Project Structure: Standard src-layout
- **Decision**: Mandate a top-level namespace structure like `src/aegisos/`, with test code in the `tests/` directory.
- **Rationale**:
  - Secures a unique top-level namespace, completely preventing import conflicts with common directory names like `core` or `agents` (namespace pollution).
  - Aligns with modern Python build tool (like `uv`) best practices, ensuring consistency between local testing and the final packaged production environment.

## 7. Hybrid Architecture: Python AI Logic + Go Network Sidecar
- **Decision**: Use Python for the main process to handle Agent scheduling and AI logic, while implementing the underlying P2P network stack in Go (based on `go-libp2p`), running as a Sidecar process.
- **Rationale**:
  - **Performance**: Go is more stable and has lower resource usage than Python when handling high-concurrency P2P state machines, DHT addressing, and NAT traversal.
  - **Ecosystem**: `libp2p` is much more mature in Go than in the Python community.

## 8. Deployment Strategy: Binary Bundling
- **Decision**: Package cross-compiled Go binaries directly into the `bin/` directory of the Python Wheel.
- **Rationale**:
  - **Seamless Experience**: Users can get complete cross-machine collaboration capabilities via `uv add aegisos` or `pip install`, without manually installing a Go environment or configuring complex P2P nodes.
  - **Process Management**: The Python launcher automatically triggers the corresponding platform-specific Sidecar process and manages its lifecycle.

## 9. Agent Behavior: Autonomous Reaction and NOOP Mechanism
- **Decision**:
  - `AACPAgent` automatically triggers a `think()` loop via `handle_message` upon receiving a message.
  - `AACPResponse.receiver` can be `None`, representing the Agent's decision to take no further action (NOOP).
- **Rationale**:
  - **Autonomy**: Reduces explicit `think` calls, giving Agents native reactivity.
  - **Loop Prevention**: Breaks infinite message loops between an Agent and themselves or others via an explicit NOOP state.

## 10. System Kernel: Dynamic Spawning
- **Decision**: The `system@local` agent in `AegisDispatcher` dynamically instantiates Agent classes via the `SPAWN` intent, rather than just registering simple callback functions.
- **Rationale**:
  - **Object-Oriented Management**: Ensures all spawned Agents have complete cognitive and memory management capabilities, rather than being isolated functions.
  - **Lifecycle Management**: Combined with the `TERMINATE` intent, it implements automated agent onboarding and destruction processes.

## 11. Agent Architecture: Protocol vs. Implementation (Class)
- **Decision**: Adhere to the "Protocol First" principle. Any entity following the AACP protocol (capable of sending/receiving `AACPMessage`) is viewed as an Agent by AegisOS.
- **Rationale**:
  - **Polymorphism**: The system must support multiple types of implementations, not just LLM-driven ones.
  - **Efficiency & Cost**: Introduces functional Agents (Stub/Functional) to handle high-frequency system tasks (e.g., monitoring, heartbeats) without consuming tokens and with low latency.
  - **Determinism & Safety**: Critical system paths (e.g., resource destruction) should be executed by functionally deterministic Agents.
  - **Heterogeneous Integration**: Lays the foundation for future Phase 5 Go-Sidecar cross-language communication.

## 12. Resource Decoupling: Introducing AgentFactory
- **Decision**: Extract Agent instantiation logic from `AegisDispatcher` and move it to an independent `AgentFactory`.
- **Rationale**:
  - **Kernel Decoupling**: Dispatcher only handles message routing, while Factory handles object construction, following the Open-Closed Principle.
  - **Dynamic Extension**: Allows registering new Agent types at runtime without modifying the core Dispatcher code.

## 13. Semantic Layer Decoupling: Intent vs. Action
- **Decision**: `AACPIntent` only describes communication primitives (Request/Reply/Inform), while specific business operations (e.g., code execution, file I/O) are encapsulated in `payload["action"]` and normalized by the `AACPAction` enum.
- **Rationale**:
  - **Orthogonality**: Prevents the `AACPIntent` enum from bloating indefinitely as business capabilities increase.
  - **Generality**: Makes AACP a true "meta-protocol" adaptable to any type of Agent collaboration, not just code execution.
  - **Scalability**: Enables finer-grained permission control and routing via namespaced Actions (e.g., `core.exec.*`).

## 14. Economic Interaction: Off-chain Coordination + On-chain Settlement

- **Decision**:
  Use AACP for off-chain task coordination and negotiation, while using blockchain networks for settlement.

- **Rationale**:
  - Efficiency: Avoid high on-chain costs for every interaction
  - Flexibility: Allow complex multi-agent workflows off-chain
  - Trust: Use on-chain settlement for final value exchange

## 15. Responsibility Boundary: context_pointer vs. payload
- **Decision**: Strictly separate transient execution data from persistent cognitive state.
    - `payload`: Transient input for the immediate action.
    - `context_pointer`: Cognitive state, including execution pointers (`current_task`) and state references (`uri`).
- **Rationale**:
    - Minimizes token usage by passing references instead of full data.
    - Enables "zero-thinking" execution for workers.
    - Ensures persistent state is maintained in the workspace, not the message history.

## 16. Role Separation: Coordinator-Worker Pattern
- **Decision**: Decouple the "Decision Power" from "Execution Power" (Phase-1 design).
    - Coordinator: Owns the plan and decides "what to do".
    - Worker: Owns the execution of the "current task" and decides "how to do it".
- **Rationale**:
    - **Stability & Debuggability**: In Phase-1, strict separation ensures determinism and makes the collaboration loop easier to trace.
    - **Planning Simplicity**: Prevents planning complexity from overwhelming individual execution agents.
    - **Scalability**: Enables scalable multi-agent collaboration with a centralized or federated planning layer.
- **Roadmap**: Future iterations will evolve this into **Hierarchical Delegation**, allowing for controlled decentralization of decision-making.
